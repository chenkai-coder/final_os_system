# MYFS 核心数据结构

## 一、文件物理结构 —— 多级索引

每个文件对应一个 inode（`myfs_inode_t`，定长 256 字节），其中用 **12 个直接块 + 1 个一级间接块 + 1 个二级间接块** 来记录文件数据所在的所有物理块号。

```
inode (256 bytes)
├─ direct[0..11]     12 个直接块  → 直接指向数据块        (12 × 4KB = 48KB)
├─ indirect1          1 个一级间接块 → 块内 1024 个指针    (1024 × 4KB = 4MB)
└─ indirect2          1 个二级间接块 → 1024 个一级块       (1024² × 4KB = 4GB)
                                         每个再 1024 个指针
```

核心 inode 结构（`os_system/include/inode.h`）：

```c
typedef struct myfs_inode {
    myfs_ino_t inode_id;          // inode 编号
    uint16_t   type;              // FREE=0, FILE=1, DIR=2, SYMLINK=3
    uint16_t   mode;              // 权限位 (0644/0755)
    uint32_t   uid;               // 所有者
    uint32_t   gid;               // 所属组
    uint64_t   size;              // 文件逻辑大小（字节）
    uint32_t   block_count;       // 已分配物理块数
    uint32_t   link_count;        // 硬链接数
    uint32_t   open_count;        // 打开计数
    uint64_t   atime, mtime, ctime, crtime;  // 时间戳

    // ★ 核心：三级块指针
    myfs_block_t direct[12];      // 直接块
    myfs_block_t indirect1;       // 一级间接块
    myfs_block_t indirect2;       // 二级间接块

    uint32_t   checksum;          // 校验和
    uint8_t    reserved[120];     // 预留，保证 inode 定长 256B
} myfs_inode_t;
```

逻辑块号 → 物理块号的映射过程（`os_system/include/block_map.h`）：

```
逻辑块号 < 12          → direct[logical] 直接命中
逻辑块号 ∈ [12, 1035]   → indirect1 块中第 (logical-12) 项
逻辑块号 ∈ [1036, ~1M]  → indirect2 → 第 i 个一级块 → 第 j 项
```

相关常量：
- 块大小 `MYFS_BLOCK_SIZE = 4096`（4KB）
- 每间接块存 `MYFS_POINTERS_PER_BLOCK = 1024` 个指针（4096÷4）
- 最大逻辑块 ≈ 1,048,612，支持文件最大约 **4GB**

## 二、目录结构 —— 含目录项数组的普通文件

目录在物理上是一种特殊的文件（`type = MYFS_INODE_DIR`），其数据块中存放的是 **目录项数组**。每条目录项由 inode 编号和文件名组成。

目录项结构（`os_system/src/vfs_upper.cpp:53-57`）：

```c
#define DIRSIZ 14    // 文件名最长 13 字符（含 '\0'）

#pragma pack(push, 1)
struct vfs_dirent {
    unsigned short d_ino;       // 文件/子目录的 inode 编号
    char           d_name[DIRSIZ];  // 文件名，d_ino=0 表示空闲项
};
#pragma pack(pop)
// sizeof(vfs_dirent) = 16 字节，每 4KB 块可存 256 个目录项
```

一个目录的数据块内容示意：

```
块 0:
┌────┬──────────────┬────┬──────────────┬────┬──────────────┬...
│ 0  │ "."           │ 0  │ ".."          │ 5  │ "home"        │
├────┼──────────────┼────┼──────────────┼────┼──────────────┼...
│ 8  │ "usr"         │ 12 │ "test.c"       │ 0  │ ""            │
└────┴──────────────┴────┴──────────────┴────┴──────────────┴...
```

目录初始化伪代码（`vfs_init_dir`）：

```
vfs_init_dir(dir_inode, parent_inode):
    entries = 分配一整个 4KB 块，全部清零
    entries[0] = { d_ino: dir_inode,    d_name: "."  }   // 自身
    entries[1] = { d_ino: parent_inode, d_name: ".." }   // 父目录
    将该块写入 dir_inode 的第 0 逻辑块
```

路径查找 `namei()` 流程：

```
namei("/home/usr/test.c"):
    inode = iget(0)                       // 从根 inode 开始
    for part in ["home", "usr", "test.c"]:
        entry = 在当前 inode 的所有数据块中遍历 vfs_dirent[]
                找到 d_name == part 的那一项
        inode = iget(entry.d_ino)         // 进入下一级
    return inode
```

## 三、空闲盘块组织 —— 成组链接法

数据块的分配和回收采用 **成组链接法**。超级块中保存当前可直接分配的一组空闲块号（最多 100 个），这些块号用完后从磁盘上的组管理块读入下一组。

### 超级块中的当前组

```c
#define MYFS_FREE_GROUP_SIZE 100

typedef struct myfs_superblock {
    // ... 其他字段 ...
    uint32_t    free_group_count;               // 当前组剩余块数
    myfs_block_t free_group[MYFS_FREE_GROUP_SIZE]; // 当前组 100 个空闲块号
    // ...
} myfs_superblock_t;
```

### 磁盘上的组管理块

当一个空闲块被用作"链接到下一组"的桥梁时，它内部存储：

```c
typedef struct myfs_free_group_block {
    uint32_t    count;                          // 下一组有多少个空闲块
    myfs_block_t blocks[MYFS_FREE_GROUP_SIZE];  // 下一组 100 个块号
    uint32_t    checksum;                       // 校验
} myfs_free_group_block_t;
```

### 分配流程

```
myfs_block_alloc():
    if 超级块.free_group_count == 0:
        return NO_SPACE

    取 free_group[count - 1] → 这就是要分配的块
    count--

    if count == 0（刚拿走了这组最后一个）:
        读取刚分配的块（它其实是下一组的"管理块"）
        把它的 blocks[] 和 count 复制到超级块的 free_group[]
        这个块本身已经被"分配出去"，不再空闲

    return 分配的块号
```

### 释放流程

```
myfs_block_free(block_id):
    if 超级块.free_group_count < 100（当前组未满）:
        free_group[count] = block_id
        count++
    else（当前组已满）:
        把超级块当前的 free_group[] 写入 block_id（block_id 变成新管理块）
        清空 superblock.free_group[]
        free_group[0] = block_id   // 新组第一个就是刚释放的块
        count = 1
```

### inode 的管理

inode 不参与成组链接，而是用独立的 **inode 位图**（bitmap）管理，位于磁盘 inode 位图区。

## 磁盘整体布局

```
┌────────────────────────────────────────────────────────────┐
│ Block 0                  │ 超级块 (myfs_superblock_t)       │
├──────────────────────────┼─────────────────────────────────┤
│ Block 1 ~ x              │ inode 位图（每个 inode 占 1 bit） │
├──────────────────────────┼─────────────────────────────────┤
│ Block x+1 ~ y            │ inode 表（每个 inode 256 字节）    │
├──────────────────────────┼─────────────────────────────────┤
│ Block y+1 ~ z            │ journal 日志区（已预留，未实现）    │
├──────────────────────────┼─────────────────────────────────┤
│ Block z+1 ~ 32767        │ 数据区（文件数据 + 目录项 + 组管理块）│
│                          │ 通过成组链接法管理空闲块            │
└──────────────────────────┴─────────────────────────────────┘
    总块数 = 32768（format 时指定），块大小 = 4096 字节
    磁盘镜像总大小 = 128 MB
```

## 系统特色

本系统的文件物理组织采用经典的多级索引方案，每个 inode 内嵌 12 个直接块指针、1 个一级间接块指针和 1 个二级间接块指针。对于几 KB 到几十 KB 的小文件，直接块足以覆盖其全部数据，一次磁盘访问即可定位目标块，无需额外读取索引块；对于中等大小的文件，一级间接块通过一个 4KB 的索引页即可管理 1024 个数据块，覆盖 4MB 的范围；对于更大的文件，二级间接块将容量进一步扩展到约 4GB，远超磁盘镜像本身的 128MB 物理容量。这种三级递进的映射结构带来的另一个重要特性是稀疏文件支持：文件可以在任意偏移处写入数据，中间未经写入的逻辑块不会分配物理磁盘空间，形成所谓的"空洞"，读取空洞时底层直接返回全零而无需触发磁盘 I/O。这一设计不仅节省了存储空间，也使得诸如虚拟机磁盘镜像、数据库预分配文件等场景能够在用户态文件系统中得到自然的支持。

空闲数据块的管理没有采用常见的位图方式，而是实现了 UNIX 经典文件系统中使用的成组链接法。超级块中常驻一个大小为 100 的空闲块号数组，文件系统运行时，绝大多数块分配操作只需从该数组末尾取出一个块号即可完成，块释放操作也只需将块号追加回数组末尾，整个过程不涉及任何磁盘 I/O。仅当数组耗尽（分配时取完最后一块）或数组满溢（释放时已无空位）时，系统才触发一次磁盘读写，从磁盘上的组管理块中读入下一组 100 个空闲块号，或将当前这组空闲块号写入磁盘腾出空间。这意味着每 100 次分配或释放操作才产生一次 I/O，且单次 I/O 即为批量操作，无需像位图那样逐位扫描查找空闲位。成组链接法的另一个巧妙之处在于，被用作组管理块的那些磁盘块本身也是空闲块，当该组被读入超级块后，这个管理块就可以作为普通数据块分配出去，不会造成任何空间浪费。inode 的空闲管理则与数据块分开，使用独立的 inode 位图，以比特位标记每个 inode 是否被占用，两者的管理机制互不干扰，各取所长。

本系统完整实现了 POSIX 标准的硬链接机制，并通过 link 指令向用户暴露该功能。硬链接允许多个目录项指向同一个 inode，它们在文件系统中是完全等价的入口，不存在主次之分。inode 结构中的 link_count 字段记录了指向该 inode 的硬链接总数，当用户执行 link 操作时，系统在目标目录中创建一个新的目录项，将其 d_ino 指向源文件的 inode，同时将 link_count 加一；当执行 unlink（删除）操作时，仅移除指定的目录项并将 link_count 减一，只有当 link_count 降为零时才真正释放 inode 及其占用的数据块。这一设计使得同一个文件可以以不同的名字出现在不同的目录下，且所有名字具有同等地位，删除其中一个不会影响其他入口对文件的访问，非常适合共享文件、版本回退等应用场景。

本系统实现了完整的多用户权限管理机制，每个 inode 都记录了所有者用户 ID（uid）、所属组 ID（gid）以及 9 位传统 UNIX 权限位。权限位分为三组，分别控制所有者、同组用户和其他用户的读、写、执行权限，例如 0644 表示所有者可读写、同组和其他用户只读。系统在每次文件访问时都会根据当前进程的有效用户 ID 和目标 inode 的权限位进行权限校验，只有通过校验才允许执行相应操作。此外，超级用户（uid 为 0）拥有特权，可以绕过权限检查执行任意操作。这套权限系统与 VFS 层紧密集成，为后续扩展 ACL 等高级权限模型预留了空间，使得本文件系统能够在多用户环境下安全运行。

本系统设计了完整的故障检测与恢复机制以保障文件系统的可靠性。fsck 模块实现了全面的一致性检查，包括超级块字段校验、inode 位图与 inode 表的一致性、数据块引用的合法性、重复引用检测、成组链接法空闲链的完整性、以及数据区泄漏块的扫描等十余项检查内容，并输出详细的错误统计报告。日志系统采用 write-ahead logging 机制，在修改元数据块前先将目标块号和新内容写入 journal 区，标记事务为 committed 状态后再写回真正的目标块，若程序在事务提交后、完成前崩溃，下次 mount 时 journal_recover 会自动检测并重放未完成的事务，将文件系统恢复到一致状态。这套日志机制保护的对象包括超级块、inode 表、inode 位图等关键元数据，确保即使系统意外断电或崩溃，元数据也不会处于不一致的中间状态，实现了类 ext4 日志文件系统的可靠性特性。

系统在整体架构上采用了清晰的三层分层设计。最底层为物理层，包含磁盘模拟模块、块缓存模块、inode 管理模块和块分配模块，这一层只处理块号、inode 编号和缓冲区，完全不感知文件名、路径或用户命令等上层概念。中间层为块映射与读写层，负责将文件的逻辑块号翻译为物理块号，处理直接块、一级间接块和二级间接块的遍历与分配，并在此基础上提供基于偏移量的读写接口。最上层为虚拟文件系统层，实现了路径解析、目录遍历、文件描述符管理、用户权限系统以及完整的 shell 命令行接口，直接对应用户的操作语义。三层之间通过明确的函数接口传递数据，每一层都可以独立编译和测试。此外，系统还配套了一个基于 PyQt5 的图形化调试终端，通过子进程管道与 C++ 后端通信，能够实时展示磁盘块热力图、目录树结构和 inode 详细信息，将底层存储状态以直观的可视化方式呈现出来，对理解和调试文件系统内部行为有很大帮助。

## 存在问题

日志模块目前仅有接口声明和磁盘布局上的预留空间，尚未实现实际的日志写入与回放机制。超级块中虽然维护了文件系统 CLEAN 和 DIRTY 的状态标记，但一旦发生崩溃或非正常断电，该标记只能告知"上次未正常卸载"，无法恢复到崩溃前的一致性状态，元数据可能存在部分写入、块分配信息丢失等问题。

另一个不足是目录查找采用线性扫描方式。目录的数据块中存放的是一个紧凑的目录项数组，查找某个文件名时，需要从头遍历该目录的所有已分配数据块，逐一比对目录项中的文件名直到命中或扫描完毕。当目录下文件数量较少时影响不大，但在文件数量较多的场景下，每次路径解析都可能产生可观的查找开销，缺少哈希或树形索引来加速目录检索。

此外，文件名长度目前限制为最多 13 个字符（DIRSIZ 定义为 14，包含结尾的空字符）。这一限制源于目录项结构的设计，每个目录项固定占用 16 字节，其中文件名数组占 14 字节。虽然对于教学演示和简单应用场景足够使用，但相比现代文件系统动辄 255 字节的文件名长度上限，这一约束在处理具有描述性长文件名的实际文件时会显得不便，需要用户在导入文件时手动截断或重命名。

## 分工说明

本项目由三人协作完成。第一人负责文件系统底层物理层的全部实现工作，包括磁盘镜像的模拟与读写接口、原始文件操作封装、超级块结构的初始化与管理、inode 的分配释放与读写、成组链接法空闲块管理、多级索引块映射、块缓存机制、磁盘布局计算以及物理层 API 的统一封装。这些模块构成了文件系统的存储基础，向上层提供块级别的读写能力和元数据管理能力，是整个系统可靠运行的基石。

第二人负责上层虚拟文件系统的实现、命令行与图形界面的开发以及崩溃恢复机制的集成。在 VFS 层实现了路径解析、目录遍历、文件描述符管理、用户权限校验、硬链接支持等核心功能，并在此基础上开发了完整的命令行接口，涵盖 format、mount、login、mkdir、create、open、read、write、link、fsck、recover 等数十条命令。同时基于 PyQt5 实现了图形化调试终端，提供磁盘块热力图、目录树可视化、实时状态同步等功能。崩溃恢复方面实现了 write-ahead logging 日志系统和 fsck 一致性检查模块，保障系统在异常断电后的数据完整性。

第三人负责整个项目的测试工作，编写了覆盖各模块的测试程序。测试范围包括磁盘布局计算、物理块读写、超级块字段校验、inode 位图与 inode 表一致性、成组链接法空闲块分配与释放、块映射的正确性、缓存命中率与淘汰策略、日志系统的写入与回放、文件系统一致性检查、挂载与卸载流程等。通过系统化的测试验证了底层物理层、中间映射层和上层 VFS 层的正确性，确保各模块在边界条件和异常情况下的行为符合预期，为项目的稳定交付提供了质量保障。
